# Quest Agent 面试演讲稿

> 总时长 10-12 分钟，每页讲解时间已标注。后附 15 个高频面试问答。

---

## 第 1 页：封面（10 秒）

> "大家好，今天我要分享的项目叫 Quest Agent——一个基于多 Agent 协作的 LLM 故事驱动目标追踪系统。这是我在 2026 年 3-4 月独立完成的个人项目。"

---

## 第 2 页：项目背景（1 分钟）

> "首先说一下为什么要做这个项目。
>
> 我们日常用的打卡类 App 有个核心问题——枯燥。完成一个任务，就是数字从 0 变成 1，跳过一天也没有任何代价。长期坚持率非常低。
>
> Quest Agent 的思路是：把你的现实目标——比如'每天跑步 5 公里'——映射成一段奇幻冒险故事。你完成任务，角色就升级、获得装备、剧情正向推进；你跳过任务，角色就受伤、遭遇危机。用叙事的代入感来激励你坚持。
>
> 这个系统有 5 个协作智能体、7 个向量数据集合、4 层解耦架构，以及 13 个单元测试保障核心逻辑。"

---

## 第 3 页：系统架构（1 分钟）

> "整体架构分为四层。
>
> 最上面是 **CLI 层**，用 Click 做命令行交互，Rich 做终端美化渲染。
>
> 第二层是 **Agent 层**。GameMaster 是编排器，它自己不调用 LLM，只负责协调下面四个领域 Agent——TaskPlanner 负责任务分解，StoryTeller 负责叙事生成，Evaluator 负责进度评估，WorldBuilder 负责世界构建。它们都继承自 BaseAgent，共享 think、think_json、think_with_tools 三种 LLM 调用方式。
>
> 第三层分成两部分：**LLM 层**用 LangChain 的 ChatOpenAI 和 LCEL 做链式调用；**Memory 层**用 ChromaDB 向量数据库做持久化和语义检索。
>
> 最底层是 **Pydantic v2 的领域模型**和**工具定义**。
>
> 各层之间职责隔离，可以独立测试和替换。"

---

## 第 4 页：多 Agent 编排（1.5 分钟）

> "先看左边。GameMaster 采用的是 **Orchestrator 模式**——确定性编排，不是 LLM 自己决定下一步调谁。调用谁、什么顺序、传什么参数，全是代码写死的。这样做的好处是调用链路可预测、成本可控、调试简单。
>
> BaseAgent 是抽象基类，提供三个模板方法：
> - think 是纯文本输出，底层链是 `Prompt | LLM | StrOutputParser`；
> - think_json 强制 JSON 输出，用了 `JSON Mode + JsonOutputParser`；
> - think_with_tools 是多轮工具调用循环。
>
> 子 Agent 只需要实现一个 `run` 方法就能接入整个系统。这是模板方法模式。
>
> 右边展示了两个具体流程。新游戏是 4 次 LLM 调用，每日签到是 2 次 LLM 调用加 2 次语义搜索。次数固定、可预测，不像自由 Agent 那样可能失控。"

---

## 第 5 页：LCEL 链（1.5 分钟）

> "LCEL 是 LangChain Expression Language，本质上是**管道模式**。
>
> 看左边的代码。think_json 的核心就四步：构建 Prompt 模板、绑定 JSON 模式、用管道符 `|` 串成链、异步执行。
>
> 右边是这条链的执行过程。kwargs 传入后，ChatPromptTemplate 做模板渲染，把变量替换进去；然后发给 ChatOpenAI 调用 API——如果主模型失败，with_fallbacks 会自动切到备用模型；API 返回 JSON 字符串后，JsonOutputParser 解析成 Python dict。
>
> 关键设计点是：每个组件——Prompt、LLM、Parser——都实现了 `Runnable` 接口（有 `invoke` 和 `ainvoke`），整条链本身也是一个 Runnable。这是组合模式，可以任意嵌套。"

---

## 第 6 页：ReAct 工具调用（1 分钟）

> "工具调用采用的是 **ReAct 模式**，就是'推理 + 行动'。
>
> 设计上有一个重要决策：我**没有使用 LangChain 的 AgentExecutor**，而是手写了工具调用循环。原因是 AgentExecutor 封装度太高，错误处理和中间状态不可控。手写循环可以精确控制最大轮次、自定义降级策略。
>
> 工具定义用了 `StructuredTool.from_function`，它会通过 Python 的类型注解自动推断参数 Schema。比如 `sides: int = 20` 这个注解，Pydantic 会把它转成 JSON Schema，再转成 OpenAI 的 tools 参数格式。
>
> 还有一个设计细节：纯业务函数 `roll_dice()` 和 LangChain 包装 `create_dice_tool()` 是分离的。测试的时候直接测业务函数，不需要 mock LangChain。
>
> 右边是一个具体例子：LLM 收到'判定翻越城墙'，自己决定调用 roll_dice，系统执行后把结果反馈回去，LLM 继续推理生成叙事。"

---

## 第 7 页：RAG 向量检索（1.5 分钟）

> "这是我认为项目中技术含量最高的部分。
>
> 原来用 SQLite 的时候，上下文就是取最近 2 章。这有一个问题——如果第 1 章提到了一个伏笔，到了第 8 章才需要回收，中间全丢了。
>
> 改用 ChromaDB 后，章节全文和关键线索都会被 embedding 成向量存储。生成新章节时，ContextAssembler 把当前角色状态和世界状态拼成一个查询文本，发给 embedding API 得到向量，然后在 ChromaDB 里做余弦相似度搜索，找到最相关的 3 个历史章节和 5 条未解决线索。
>
> 这就是一个轻量级的 RAG 流程：Retrieve 从向量库检索，Augment 组装成上下文，Generate 让 LLM 生成新章节。
>
> ChromaDB 有两种查询方式：get 是纯 metadata 过滤，类似 SQL 的 WHERE，不需要 embedding；query 是语义相似度搜索，需要 embedding。结构化数据用 get，文本数据用 query，各取所长。"

---

## 第 8 页：完整数据流（1.5 分钟）

> "用一个具体例子走一遍每日签到的完整流程。
>
> 第一步，用户在 CLI 里汇报 3 个任务的完成状态。
>
> 第二步，Evaluator 调用 LLM，评估出完成度 60%、结局类型 setback、获得 70 XP。这里有一个 fallback 机制——如果 LLM 返回的 JSON 缺少 completion_pct 字段，Evaluator 会用纯代码 `_calc_completion` 计算，不会崩溃。
>
> 第三步是纯代码：给角色加 XP、扣血、加属性、更新连续打卡天数。
>
> 第四步，ContextAssembler 做语义检索，从 ChromaDB 找相关章节和线索。
>
> 第五步，StoryTeller 拿到上下文和评估结果，调用 LLM 生成新章节。60% 完成度加上 setback 结局，LLM 会生成类似'因忽视恢复训练导致膝伤'的叙事——现实行为直接驱动虚拟剧情。
>
> 第六步，新章节存入 ChromaDB 被 embedding，下次就能被语义检索到；新伏笔存入 key_facts 集合。
>
> 整个链路固定 2 次 LLM 调用加 2 次语义搜索，完全可预测。"

---

## 第 9 页：存储设计（1 分钟）

> "ChromaDB 里共 7 个 Collection。每条数据有两部分：document 是被 embedding 的文本，metadata 是结构化字段。
>
> 关键是 chapters 和 key_facts 这两个——它们用 query 做语义搜索，是整个 RAG 的数据基础。其他 5 个集合主要用 get 做精确查询。
>
> 序列化策略是用 Pydantic 的 model_dump_json 把整个对象转成 JSON 字符串存入 metadata 的 data_json 字段，读取时 model_validate_json 一行还原。嵌套结构比如角色的 Stats、List[Item] 都是自动递归处理的。
>
> 降级方面，如果语义搜索失败就回退到取最新几章；如果 OpenAI Embedding API 不可用，ChromaDB 会自动用内置的 MiniLM 本地模型。"

---

## 第 10 页：工程实践（1 分钟）

> "简单过一下工程实践。
>
> 设计模式方面用了 6 种：工厂模式做 LLM 构建、模板方法做 BaseAgent、门面模式做 GameMaster 编排、管道模式做 LCEL 链、组合模式做 Runnable 嵌套、充血模型让 Character 自己管理升级逻辑。
>
> 错误降级是三层防护：模型层 fallback、解析层 try-catch、业务层纯代码兜底。
>
> 测试策略是只测确定性层——模型和工具的纯逻辑——不测 LLM 输出。因为 LLM 不确定、不可复现，测了也没意义。"

> "技术栈总结在这个表格里。核心是 LangChain 做 LLM 编排、ChromaDB 做向量存储、Pydantic 做数据建模。
>
> 可扩展方向有四个：接入真实任务 API 比如 Todoist、多用户并发、LangSmith 调试、长期偏好记忆。"

# 面试问答（15 题）

---

### Q1：为什么选择多 Agent 架构而不是单个 LLM 处理所有任务？

> 单个 prompt 同时处理评估、叙事、规划，会导致注意力分散、输出质量下降。拆分后每个 Agent 的 system prompt 更聚焦，输出更稳定。而且各 Agent 可以独立迭代提示词，比如我可以单独优化 StoryTeller 的叙事质量而不影响 Evaluator 的评估逻辑。

---

### Q2：GameMaster 为什么不用 LLM 做路由决策？

> 因为业务流程是确定的——新游戏就是"建世界→分任务→写章节"，签到就是"评估→更新角色→写章节"。不存在"LLM 需要动态判断下一步该调谁"的场景。硬编码编排的优势是：调用链路可预测、LLM 调用次数固定（成本可控）、出 bug 直接定位到具体 Agent。如果将来流程变复杂，比如加入"角色对话""支线选择"等动态分支，可以考虑引入 LLM 路由。

---

### Q3：为什么手写工具调用循环而不是用 AgentExecutor？

> AgentExecutor 封装度太高，三个问题：一是错误处理不可控，工具报错后 AgentExecutor 的默认行为可能不是我想要的；二是中间状态不可观测，调试困难；三是没有严格的轮次限制，可能陷入死循环。手写循环只多了 20 行代码，但获得了完全的控制权。

---

### Q4：向量数据库相比 SQLite 的具体优势是什么？举个实际例子。

> 比如第 1 章提到"主角发现了一张神秘地图"，到第 8 章需要回收这个伏笔。SQLite 方案是取最近 2 章，第 1 章的内容早就丢了。ChromaDB 方案是：当第 8 章的世界状态提到"地图上标记的位置"时，语义搜索会发现第 1 章的内容和这个 query 高度相关，自动召回。这就是语义相似度检索的价值——跨越时间找到主题相关的内容。

---

### Q5：Embedding 的过程中有性能问题吗？怎么优化？

> 目前每次写入 ChromaDB 会调用一次 Embedding API，延迟大约 100-200ms。对这个项目来说可以接受，因为写入不频繁（一天最多写几条）。如果要优化：一是可以用 ChromaDB 内置的 all-MiniLM-L6-v2 本地模型，零网络延迟；二是可以批量写入，ChromaDB 的 add/upsert 支持一次传多条；三是可以做异步写入，先返回结果再后台 embedding。

---

### Q6：如何保证 LLM 返回的 JSON 结构正确？

> 三层防护。第一层，用 `response_format={"type":"json_object"}` 让 OpenAI API 强制返回合法 JSON，语法层面不会出错。第二层，JsonOutputParser 解析，如果解析失败 try-catch 返回空 dict。第三层，业务代码里用 `.get("field", default)` 对每个字段做默认值兜底，比如 Evaluator 在拿不到 completion_pct 时，用纯代码 `sum(q.completion_pct for q in quests) / len(quests)` 计算。

---

### Q7：with_fallbacks 具体是怎么工作的？

> `ChatOpenAI(model="gpt-4o").with_fallbacks([ChatOpenAI(model="gpt-4o-mini")])` 返回一个 `RunnableWithFallbacks` 对象。它实现了和原 LLM 完全相同的接口。调用时先尝试主模型，如果抛出任何异常（网络超时、API 限流、模型不可用），自动切到备用模型重试。调用方完全无感知。

---

### Q8：Pydantic 在项目里起什么作用？

> 四个作用：一是**运行时类型校验**，LLM 返回的数据如果类型不对（比如字符串传到 int 字段），Pydantic 会自动转换或报错；二是**序列化/反序列化**，`model_dump_json` 和 `model_validate_json` 一行代码搞定对象和 JSON 的转换，对接 ChromaDB 存储；三是 **Schema 推断**，`StructuredTool.from_function` 底层就是用 Pydantic 从函数签名生成 JSON Schema；四是**充血模型**，`Character.add_xp()` 把升级逻辑封装在数据模型内部。

---

### Q9：为什么用 `str + Enum` 双继承？

> 因为 JSON 序列化时，纯 Enum 会变成 `CharacterClass.WARRIOR`，不是合法 JSON 值。继承 str 后，序列化时自动变成 `"warrior"` 字符串，和 JSON、ChromaDB metadata、LLM 输出天然兼容。读取时 `CharacterClass("warrior")` 自动还原。

---

### Q10：asyncio 在项目里解决了什么问题？

> LLM API 调用是网络 I/O，单次可能 2-10 秒。如果用同步调用，整个程序在等待 API 响应时完全阻塞。异步让程序在等待时可以做其他事。LangChain 的所有 Runnable 都提供了 `ainvoke` 异步版本，所以整个调用链都是非阻塞的。CLI 层因为 Click 不支持 async，所以用 `asyncio.run()` 包了一层。

---

### Q11：ChromaDB 的 get 和 query 有什么区别？

> get 是**纯 metadata 过滤**，不经过 embedding，类似 SQL 的 WHERE 条件。比如 `get(where={"status":"active"})` 找所有激活的任务。query 是**语义相似度搜索**，每次调用都会把查询文本转成向量，在 HNSW 索引里找最近邻。get 快、不需要 API 调用；query 有 embedding 开销但能找到语义相关的内容。

---

### Q12：工具调用时 tool_call_id 有什么用？

> OpenAI API 要求 ToolMessage 的 tool_call_id 必须和 AIMessage 里 tool_calls 的 id 一一对应。这是为了支持**并行工具调用**——LLM 可以在一轮里同时调用多个工具，每个工具结果通过 id 关联到对应的请求。如果 id 不匹配，API 会报错。

---

### Q13：ContextAssembler 的查询文本为什么用角色+世界状态拼接？

> 因为生成新章节最需要的是"与当前场景相关的历史信息"。用角色状态（职业、等级、当前 HP）和世界状态（位置、NPC、威胁）做查询，embedding 模型会理解这些语义概念，找到包含相似主题的历史章节。比如查询里有"铁拳堡"和"暗影军团"，就会把提到这些地点和威胁的旧章节排在前面。

---

### Q14：如果让你重做这个项目，有什么会改进的？

> 三个方面。第一，加入 **LangSmith** 做 prompt tracing，现在调试只能看日志，不够直观。第二，给 StoryTeller 真正启用 `think_with_tools`，让它在写故事时主动调用骰子判定，目前骰子工具注册了但实际没走 tool-use 路径。第三，考虑用 **LangGraph** 替代手动编排，特别是将来如果要加入分支选择（比如"角色遇到岔路，玩家选择走哪条"），LangGraph 的状态图更适合。

---

### Q15：这个项目的最大技术挑战是什么？

> 是 **LLM 输出的不确定性管理**。同一个 prompt 每次返回的 JSON 结构可能不一样——有时候缺字段、有时候类型不对、有时候格式错误。我花了大量精力在三层降级机制上：response_format 约束格式、JsonOutputParser 捕获异常、业务层 `.get()` 默认值加纯代码 fallback。这让系统在 LLM 返回任何"意外"输出时都能优雅降级而不是崩溃。
