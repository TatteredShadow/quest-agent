项目背景与动机

❌ 传统目标追踪的问题

  • 打卡类 App 枯燥，长期坚持率极低
  • 缺乏情感连接，完成/未完成只是冷冰冰的数字
  • 没有"后果感"——跳过一天毫无心理成本

✅ Quest Agent 的解法

  • 将现实目标映射为奇幻冒险叙事
  • 每日完成度驱动剧情正向 / 负向发展
  • 跳过任务 → 角色受伤、剧情危机
  • 连续完成 → 升级、获得装备、推动主线

核心设计 ①：多 Agent 编排

Orchestrator 模式

  • GameMaster 作为编排器，不直接生成内容
  • 每个 Agent 只做一件事 — 单一职责原则
  • Agent 间通过数据传递协作，无直接依赖

BaseAgent 基类

  • think() → 纯文本输出
  • think_json() → 结构化 JSON 输出
  • think_with_tools() → 多轮工具调用
  • 子类只需实现 run() 抽象方法

核心设计 ②：LCEL 链式调用

LangChain Expression Language — 管道模式组合 Runnable 组件

think_json 实现

async def think_json(self, template, **kw):
    # 1. 构建 Prompt
    prompt = ChatPromptTemplate.from_messages([
        ("system", self.system_prompt),
        ("human",  template),
    ])
    # 2. 绑定 JSON 模式
    json_llm = self.llm.bind(
        response_format={"type": "json_object"}
    )
    # 3. 组装 LCEL 链
    chain = prompt | json_llm | JsonOutputParser()

    # 4. 异步执行
    return await chain.ainvoke(kw)

链的执行过程

kwargs = {goal_type: "fitness", ...}
        │
        ▼
ChatPromptTemplate  → 模板渲染
        │  {goal_type} → "fitness"
        ▼
[SystemMessage(...), HumanMessage(...)]
        │
        ▼
ChatOpenAI(JSON Mode) → HTTP API 调用
        │  失败 → with_fallbacks 自动降级
        ▼
AIMessage('{"location":"铁拳堡",...}')
        │
        ▼
JsonOutputParser  → 解析为 Python dict
        ▼
{"location": "铁拳堡", ...}

每个组件都是 Runnable，实现了 invoke / ainvoke，整条链也是 Runnable — 组合模式

核心设计 ③：ReAct 工具调用循环

手动实现 vs AgentExecutor

  • 不使用 LangChain AgentExecutor（黑盒、不可控）
  • 手写循环：精确控制轮次、错误处理、日志
  • 最多 5 轮迭代，避免无限循环

核心设计 ④：向量检索增强生成（RAG）

传统方案的局限

  • SQLite + "取最近 2 章" → 只有时间顺序
  • 无法找到主题相关的历史事件
  • 叙事经常出现断裂和矛盾

ChromaDB 向量方案

  • 章节全文 → embedding 向量存储
  • 关键线索 → embedding 后可语义检索
  • 结构化数据（角色JSON）→ metadata 过滤

完整数据流：每日签到

1
用户输入：3 个任务的完成状态 → CLI 收集为 Quest 列表（status + completion_pct）
2
Evaluator → LLM：格式化任务文本 → think_json → 返回 {completion: 60%, outcome: "setback", xp: 70, comment: "忽略了恢复训练"}
fallback：LLM 解析失败时用 _calc_completion / _determine_outcome 纯代码兜底
3
更新角色（纯代码）：add_xp(70) → take_damage(5) → stats.endurance += 1 → streak_days++
4
ContextAssembler → ChromaDB 语义检索：用当前状态检索相关历史章节 + 未解决伏笔
5
StoryTeller → LLM：上下文 + 评估结果 → think_json → 生成新章节
60% 完成度 + "setback" → LLM 生成"因忽视恢复训练导致膝伤"的叙事
6
持久化：章节全文 → ChromaDB embedding 存储（下次可被语义检索）；新伏笔 → key_facts 集合

每次签到固定 2 次 LLM 调用（Evaluator + StoryTeller）+ 2 次语义搜索（章节 + 线索）

设计决策

  • Pydantic 模型序列化为 JSON 存入 metadata 的 data_json 字段
  • 读取时 model_validate_json() 反序列化还原对象
  • 无 API key 时回退到 ChromaDB 内置 ONNX 本地模型

降级策略

  • 语义搜索失败 → 回退到 load_chapters(limit=2)
  • Embedding API 不可用 → 使用 all-MiniLM-L6-v2 本地模型
  • 保证系统在任何条件下都能运行

错误降级（三层防护）

  • 模型层：主模型失败 → with_fallbacks 自动切备用
  • 解析层：JSON 解析失败 → try/except → 返回空 dict
  • 业务层：LLM 数据缺失 → 纯代码 fallback 计算

测试策略

  • 单元测试集中在确定性层（模型、工具）
  • 不测试 LLM 输出（不确定、不可复现）
  • 工具业务逻辑与 LangChain 包装分离 → 可独立测试

配置管理

  • YAML 持久配置 + dataclass 内存表示
  • API Key：优先直配 → 其次环境变量
